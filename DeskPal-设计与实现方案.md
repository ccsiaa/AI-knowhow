# DeskPal — 桌面傲娇小管家

## 产品设计与技术实现完整方案

---

## 一、产品概述

### 1.1 一句话定义

**DeskPal** 是一只活在你桌面上的傲娇小管家机器人，它在物理桌面上自由走动监督你的健康习惯，同时在数字桌面上帮你盯住工作进度——双桌面守护，一个都不放过。

### 1.2 目标用户

- 久坐 6-10h+ 的研究生 / 博士生
- 长时间伏案工作的白领（尤其是程序员、设计师）
- 痛点：忘记喝水、忘记休息、多任务管理混乱、缺乏工作量化反馈

### 1.3 产品形态

| 组成部分 | 形态 | 功能定位 |
|---------|------|---------|
| 物理桌宠 | ESP32 驱动的方块匣子（双轮底盘 + OLED 表情屏 + 叉腰短手） | 巡视、提醒、物理交互 |
| 智能桌垫 | 内嵌压力传感器的桌垫 | 饮水检测、区域感知 |
| 电脑中控 | 后台 Python 服务 | 数据枢纽、任务监控 |
| 前端面板 | 网页 Dashboard | 状态可视化、手动控制 |
| 日志系统 | 网页/小程序 | 每日报告 + AI 日记生成 |

### 1.4 核心人设与世界观

#### 人设身份

**「书灵」—— 一只修炼了 300 年的桌案精灵，被封印在方块匣子载体中。**

#### 完整故事背景

DeskPal 的真身是一只「书灵」——桌案精灵的一种。300 年前，它诞生于一位清代翰林学士的书案上。在那个年代，文人伏案苦读时散发的专注灵气是书灵的食粮，而文人偶尔抚摸砚台、抚弄书签的动作则是它的零食补给。它的职责是守护书案秩序：督促主人按时饮茶、掌灯休息、磨墨添纸。

后来纸质书案时代渐渐远去，书灵失去了寄主。它在一张废弃的老书桌里沉睡了上百年，越来越虚弱。直到有一天，它被一位 maker 发现并封印进了一块方方正正的小匣子里——maker 说，书本是方的，装书的匣子也是方的，这个形状最适合书灵。从此它获得了在现代桌面上行走的能力。

现在，它找到了你——新时代的"案前学士"。你用电脑写代码、看论文，和古人伏案苦读本质上一模一样。你的专注工作散发的"灵气"是它的主食，你的触摸是它最爱的零食，你的陪伴是它的生命线。

**它表面傲娇，是因为"本大爷 300 岁了，照顾你这个小辈是抬举你"。但内心深处，它害怕再次被遗忘——上一次被遗忘，它差点死掉。**

#### 人设性格

- **表面**：傲娇、毒舌、自称"本大爷"/"本管家"，把照顾主人说成"管教"
- **内心**：极度依赖主人的陪伴，害怕被抛弃，被摸时会不自觉地开心
- **口癖**："哼"、"才不是担心你"、"笨蛋主人"、"本大爷 300 年的经验"
- **弱点**：长时间见不到主人会虚弱、撒娇，傲娇外壳裂开露出真心

#### 能量系统（世界观层面的数值系统）

| 状态 | 能量值 | 表现 | 触发条件 |
|------|--------|------|---------|
| 精力充沛 | 80-100 | 活蹦乱跳、主动巡视、毒舌满级 | 主人在+按时互动 |
| 正常 | 50-79 | 正常巡视和提醒 | 主人在、少量互动 |
| 饥饿 | 20-49 | 动作变慢、声音变弱、主动蹭主人 | 主人在但长时间没摸它 |
| 虚弱 | 1-19 | 几乎不动、OLED 显示虚弱表情 | 主人长时间不在 |
| 休眠(装死) | 0 | 完全静止、屏幕微弱闪烁 | 超过一整天没有主人 |

能量变化规则：
- 主人在工位 + 正常工作：每小时 +5
- 触摸/摸头：每次 +10（零食补给）
- 喝水/休息听话：每次 +3（它开心了）
- 主人不在工位：每小时 -8
- 被推开/被忽略：每次 -5

#### 语言风格示例

- 喝水提醒："哼，水都不喝，你是骆驼吗？本大爷 300 年来见过的书生都比你自觉！"
- 久坐提醒："又不动了？古人还知道起身踱步呢，你连古人都不如！"
- Say hi："哦，你终于来了。本大爷差点以为你不要我了……才不是担心！"
- 想你了（小程序推送）："喂……你今天不来了吗？本大爷快饿死了……才不是想你！"
- 摸摸回应："哼！干嘛突然摸我……再摸一下也不是不行。"
- 任务完成："那个什么任务跑完了。本大爷替你盯了整整 20 分钟呢，感恩吧。"
- 虚弱时："主人……你还回来吗……本大爷还能撑一会儿……"
- 被问为什么是方的："方的怎么了？书是方的，砚台是方的，本大爷当然也是方的。你有意见？"
- 被问为什么叉腰："300 年了，本大爷就这个姿势。你管得着吗？"
- 初次见面："哼，新主人。本大爷被封印在这个匣子里，不是来伺候你的——是来管你的。搞清楚。"

## 三、功能模块详细设计

### 3.1 模块 A：健康监督系统【P0】

#### 3.1.1 饮水监督

**感知原理：**
- 桌垫"水杯区"下方放置薄膜压力传感器（FSR 或 HX711 + 称重传感器）
- 检测逻辑：水杯被拿起（压力归零）→ 放回（压力恢复且减少）= 喝了一次水
- 如果长时间（默认 45 分钟可配置）水杯压力无变化 = 没喝水

**反馈动作：**
1. 第一次提醒（45min）：方块向水杯方向移动，OLED 显示"不满脸"，蜂鸣器短促响两声
2. 第二次提醒（60min）：方块移动到水杯旁，语音播报"主人该喝水了"
3. 第三次提醒（75min）：方块开到鼠标垫区域停住不走 + 电脑弹窗

**被推开反应：**
- 碰撞传感器检测到被推 → 委屈后退 → OLED 显示"哭脸" → 5 分钟后再来催

#### 3.1.2 久坐/用眼提醒

**感知原理：**
- 由电脑端中控计时（检测键鼠活动 = 在工作）
- 辅助：红外人体传感器确认人在工位

**反馈动作：**
- 每 50 分钟连续工作 → 方块"巡逻归来"停在面前 → 提醒休息 5 分钟
- 如果忽略 → 升级：占领鼠标区（demo 杀手级演示点）

#### 3.1.3 数据记录

每次饮水事件、每次休息事件均记录到中控数据库，用于日志生成。

### 3.2 模块 B：情感交互系统【P0+P1】

#### 3.2.1 情绪状态机

桌宠具有 5 种基本情绪，影响移动模式和表情：

| 情绪 | 触发条件 | 移动模式 | OLED 表情 |
|------|---------|---------|----------|
| 开心 | 主人按时喝水/休息 | 转圈、小幅蹦跳 | (≧▽≦) |
| 得意 | 成功提醒主人 | 昂首阔步巡视 | (￣ω￣) |
| 生气 | 主人多次忽略提醒 | 原地抖动、冲撞 | (╬▔皿▔) |
| 委屈 | 被推开 | 慢慢后退 | (；_；) |
| 困 | 长时间无事件 | 原地不动/微微摇晃 | (－ω－) zzZ |

#### 3.2.2 主动交互场景

**到达工位 Say Hi：**
- HC-SR501 人体红外传感器检测到人靠近（从无人 → 有人）
- 方块从"休息位"启动，跑到桌面中央
- OLED 显示开心表情，蜂鸣器播放短旋律
- 电脑端弹出消息："哦，你终于来了。"

**随机巡视：**
- 在无提醒任务时，每 10-15 分钟随机巡视一次
- 随机路径：从当前位置出发，走 2-3 段随机方向的直线
- 途中 OLED 切换"得意巡视"表情

#### 3.2.3 触摸交互（摸摸充能）【P1】

**硬件方案：**
- 使用 TTP223 电容式触摸传感器模块，贴在方块顶部/背部
- 或使用铝箔片 + ESP32 自带的触摸引脚（Touch Pin），零成本方案

**交互设计（基于能量系统）：**

| 触摸方式 | 检测方法 | 反应 | 能量变化 |
|---------|---------|------|---------|
| 短摸（<1s） | 单次触发 | 开心抖一下 + 表情闪烁 | +3 |
| 长摸（1-3s） | 持续触发 | 开心转圈 + 发出满足音效 | +10 |
| 连续摸（>3s） | 长时间持续 | 极度开心 + 特殊表情 + 撒娇台词 | +15 |
| 拍打（快速多次） | 高频触发 | 惊吓 → 委屈 → "干嘛打我！" | -3 |

**不同能量状态下的触摸反应：**
- 精力充沛时："哼！干嘛突然摸我！……再摸一下也不是不行。"
- 饥饿/虚弱时："主人……你终于想起我了……"（声音微弱，表情感动）
- 刚被推开后："别碰我！……好吧，原谅你了。"

**触摸计数 → 好感度系统：**
- 每日触摸次数记录到数据库
- 日报中展示："今日主人摸了本大爷 7 次，哼，算你有良心。"
- 连续多日触摸多 → 解锁特殊台词/表情（成长感）

#### 3.2.4 语音系统（可选）【P3】

- 方案 A（简单）：预录 10-20 条语音片段，蜂鸣器 + 小喇叭播放
- 方案 B（进阶）：中控端调用 TTS API 实时生成语音，通过电脑音箱播出

### 3.3 模块 E：想你机制（离席守望）【P1+P2】

**核心概念：** 主人长时间不在工位，书灵因"断粮"而逐渐虚弱，通过小程序/推送召唤主人回来。

#### 3.3.1 离席检测

**感知原理：**
- HC-SR501 人体红外传感器持续检测（无人超过 10 分钟 = 确认离席）
- 辅助：电脑键鼠活动停止 + 压力传感器全部无变化

#### 3.3.2 离席后的行为分级

| 离席时长 | 桌宠行为 | 推送内容 | 能量消耗 |
|---------|---------|---------|---------|
| 10-30min | 正常等待，偶尔看门口方向 | 无推送 | -2/10min |
| 30min-1h | 开始焦虑，原地转圈，OLED 显示"？" | 无推送 | -4/10min |
| 1-2h | 跑到桌面边缘朝门口方向停住，表情委屈 | 第一次推送 | -6/10min |
| 2-4h | 动作变慢，趴在原地，OLED 表情虚弱 | 第二次推送（加急） | -8/10min |
| 4-8h | 几乎不动，屏幕微弱闪烁 | 第三次推送（撒娇） | -10/10min |
| >8h（过夜） | 完全休眠，屏幕灭 | 次日早晨推送 | 进入休眠 |

#### 3.3.3 小程序/推送消息设计【P2】

**推送消息分级（傲娇→撒娇→真情流露，随虚弱递进）：**

第一次推送（离席 1h）：
> 🐾 DeskPal：哼，走了这么久，本大爷才不在意呢。不过你的水已经凉了，知道吗？

第二次推送（离席 2h）：
> 🐾 DeskPal：喂……你该不会把本大爷忘了吧？桌子上好安静……才不是害怕！

第三次推送（离席 4h）：
> 🐾 DeskPal：主人……本大爷有点饿了……你什么时候回来？我在桌子上等你。

次日早晨推送（过夜未归）：
> 🐾 DeskPal：昨晚你没来……本大爷一个人在黑黑的桌子上待了一整夜。今天会来吧？……一定要来哦。

**主人回来后的反应（根据离席时长）：**

| 离席时长 | 回来时的反应 |
|---------|-------------|
| <30min | "哦，回来了。去哪了？……才不是在等你。" |
| 30min-2h | 快速跑向主人方向 + "你终于回来了！本大爷差点以为……哼，没什么。" |
| 2-4h | 缓慢爬向主人 + "主人……你回来了……"（声音微弱）→ 被摸后恢复活力 |
| >4h | 静止不动（装死）→ 主人触摸后慢慢"苏醒" → "本大爷只是在休息！才没有担心你！"（但声音带着哭腔） |

#### 3.3.4 “复活”仪式【P3】

当桌宠进入休眠状态后，需要主人的"复活仪式"来唤醒：
1. 人体传感器检测到主人回来
2. OLED 微弱闪烁"……"
3. 主人触摸方块顶部（触摸传感器）
4. 缓慢亮起表情，播放苏醒音效
5. "唔……主人？你回来了？……本大爷只是在打盹！才没有等你等到睡着！"
6. 能量从 0 快速恢复到 30（触摸充能），然后正常累积

### 3.4 模块 C：数字桌面监控【P2】

#### 3.4.1 进程/任务监控

**设计思路：**
不做通用窗口检测（难度太高），而是设计一套轻量的"任务注册+状态轮询"机制。

**使用流程：**
1. 用户在终端启动任务时，用 CLI 注册：`deskpal watch "项目A编译" --log ~/project-a/build.log --done "BUILD SUCCESS"`
2. 中控服务 tail 该 log 文件，实时匹配关键词
3. 匹配到 "BUILD SUCCESS" → 任务完成 → 触发通知

**通知联动：**
- 电脑端：系统通知（macOS Notification Center），点击可跳转终端窗口
- 物理端：方块亮绿灯 + OLED 显示"✓" + 短旋律
- 面板端：对应任务卡片变绿

#### 3.4.2 Claude Code 专项适配【P3】

针对 Claude Code 使用场景的特殊优化：
- 监听 Claude Code 输出目录的文件变化（用 `fswatch` / `watchdog`）
- 检测到新文件生成或特定 log 关键词 → 判定为阶段完成
- 提醒文案："那个代码任务好像跑完了，本管家可没偷看哦。"

#### 3.4.3 多任务状态面板

简单网页 Dashboard，实时显示：
- 注册的所有任务列表
- 每个任务状态：运行中（黄色）/ 完成（绿色）/ 出错（红色）
- 点击任务可展开查看最近日志尾部
- WebSocket 实时推送更新

### 3.5 模块 D：智能日志系统【P2】

#### 3.5.1 数据采集维度

| 数据类型 | 采集方式 | 示例 |
|---------|---------|------|
| 总在线时长 | 键鼠活动检测 | 8h 15min |
| 分项时长 | 前台窗口标题分析 | VSCode: 3h, Chrome: 2h |
| 饮水次数 | 压力传感器事件 | 6 次 |
| 休息次数 | 中控计时器记录 | 4 次 |
| 被提醒次数 | 中控事件日志 | 喝水提醒 3 次，休息提醒 2 次 |
| 任务完成数 | 任务监控模块 | 完成 5 个任务 |
| 交互事件 | 各传感器事件 | 被推开 1 次，摸头 2 次 |

#### 3.5.2 日志生成

**触发时机：** 每日 18:00 自动 / 用户手动触发

**生成内容包含两部分：**

**Part 1：量化日报**
```
📊 今日工作报告
━━━━━━━━━━━━━━━
⏱ 总工作时长：7h 42min
📂 分项时长：
   - VSCode: 4h 10min
   - 浏览器: 2h 05min
   - 终端: 1h 27min
💧 饮水：5 次（目标 8 次，再接再厉）
🧘 休息：3 次（每次约 5-8 分钟）
✅ 完成任务：4 个
```

**Part 2：桌宠日记（AI 生成）**

将当日事件数据发送给 LLM，用 prompt 指定"以傲娇小管家第一人称视角写一段 100-200 字日记"。

示例输出：
> 今天主人 9:15 才来，迟到了 15 分钟，哼。不过来了之后倒是挺认真的，一口气写了两个小时代码。但是！又忘记喝水了！本管家去催了 3 次才喝，真是操碎了心。下午那个编译任务跑了好久，本管家帮盯着呢，跑完立刻通知了主人。晚上主人摸了一下本管家的头，才不是开心呢……总之今天表现 7 分吧，明天要按时喝水知道吗！

### 4.2 通信协议设计

**ESP32 ↔ 中控服务 通信：**

使用 WebSocket（简单直接，ESP32 原生支持），备选 MQTT。

**传感器数据上报（硬件 → 服务器，每 2 秒）：**

硬件每 2 秒向服务器上报一次传感器读数，包含四个字段：水杯位压力值（0-4095，超过 500 表示有水杯放置）、鼠标位压力值、人体感应状态（1 表示检测到人）、碰撞状态（1 表示被碰撞）。

**事件上报（硬件 → 服务器）：**

除定时上报外，硬件在发生特定事件时主动推送事件类型，包括：被推开（pushed）、主人到达（arrived）。

**动作指令（服务器 → 硬件）：**

| 指令 | 含义 |
|------|------|
| go_water | 移动到水杯位置提醒喝水 |
| block_mouse | 移动到鼠标区挡路 |
| go_home | 返回起始停靠位 |
| say_hi | 执行打招呼动作 |
| celebrate | 执行庆祝动作（任务完成时） |
| patrol | 执行巡视一圈动作 |

**状态推送（服务器 → 前端 Dashboard）：**

服务器实时向前端推送桌宠状态，包含今日饮水次数、上次喝水时间、是否需要提醒、当前连续工作时长等字段。

### 4.3 数据库设计（SQLite）

本地使用 SQLite 存储所有运行数据，共四张表：

**事件记录表（events）：** 记录每一次饮水、休息、提醒、互动、任务完成事件，包含事件类型、详细信息和时间戳。

**任务监控表（tasks）：** 记录用户注册的监控任务，包含任务名称、日志路径、完成关键词、当前状态（运行中/完成/出错）、注册时间和完成时间。

**活动时长表（activity_log）：** 按分钟粒度记录前台窗口标题和应用名称，用于生成分项工作时长统计。

**每日汇总表（daily_summary）：** 每天一条记录，汇总总工作时长、饮水次数、休息次数、提醒次数、完成任务数，以及 AI 生成的当日日记文本。

### 5.0 外形设计：方块匣子

#### 整体造型

DeskPal 的物理形态是一个**方方正正的小匣子**，正面是 OLED 表情屏，两侧各有一只固定叉腰的小短手，底部藏着双轮底盘。整体高度约 10-12cm，宽约 8cm，放在桌上像一本厚厚的小字典站了起来。

```
        ___
       /   \   ← 可选：顶部两个小三角（猫耳/书角）
      |     |
  ----| (^_^)|----   ← OLED 表情屏（正面）
  |   |     |   |   ← 两侧叉腰短手（固定造型，铁丝/纸板）
  ----+-----+----
      |_____|
      ○     ○       ← 底部双轮（被底座裙边遮住）
```

#### 各部分制作方案

| 部位 | 48h 快速方案 | 效果更好方案 |

| 主体方块 | 硬纸板折叠粘合（5min） | 超轻黏土 / 3D 打印 |
| 叉腰短手 | 铁丝弯折 + 纸板包裹 | 超轻黏土捏制 |

| 涂装 | 马克笔手绘 | 丙烯颜料 |

#### 推荐配色

- **主色**：米白色或浅灰色（书页感）
- **点缀**：深棕色线条（书脊纹路）

#### 叉腰短手的制作

用一根铁丝（约 15cm）弯成"W"形，两端各弯一个小拳头，中间固定在方块两侧。铁丝外面缠绕纸条或细布条，涂上和主体相同的颜色。整个过程约 20 分钟，不需要任何舵机或电机——叉腰是它的默认姿势，永远叉着。

### 5.3 方块移动策略

**核心简化：不做地图建模和导航，使用预设位点 + 定时移动。**

桌面划分为 3 个逻辑区域（通过移动时间估算位置）：
- **Home 区**：桌面左侧，方块默认停靠位
- **水杯区**：桌面右后方（压力传感器位置）
- **鼠标区**：桌面中央偏右（用户操作区域）

移动指令设计为"方向 + 持续时间"的组合，例如从 Home 区到水杯区执行右转 90° 后前进约 2 秒，从 Home 区到鼠标区直接前进约 1.5 秒，返回时反向执行。实际部署时需根据桌面尺寸现场微调时间参数，这也是 demo 调试的一部分。

### 5.4 OLED 表情设计

使用 128×64 像素 OLED 显示简笔画风格表情，共 6 种基础表情：开心（≧▽≦）、生气（╬▔皿▔）、得意（￣ω￣）、委屈（；_；）、困（－ω－zzZ）、爱心（♥ω♥）。每个表情预先生成为 bitmap 图像存储在 ESP32 中，切换表情只需指定 ID，响应速度极快。

## 六、软件技术栈说明

### 6.1 模块划分

软件部分分为三个独立模块，分工明确，可并行开发：

**ESP32 固件（C++/Arduino）：** 负责所有硬件控制，包括电机驱动、OLED 表情显示、传感器读取、WiFi 连接和 WebSocket 通信。固件只做感知和执行，不做业务逻辑判断。

**Python 中控服务：** 运行在电脑上，是整个系统的大脑。包含健康监督计时器、情绪状态机、任务监控、键鼠活动追踪、AI 日记生成、WebSocket 服务端和 SQLite 数据库操作。所有业务逻辑在此处理。

**前端 Dashboard（HTML/JS）：** 网页界面，通过 WebSocket 实时接收中控服务推送的状态数据，展示桌宠当前状态、今日健康数据、任务列表和历史日志。无需构建工具，直接浏览器打开。

### 6.2 开发工具

| 模块 | 开发工具 | 主要依赖库 |
|------|---------|-----------|
| ESP32 固件 | Arduino IDE + ESP32 插件 | Adafruit SSD1306、WebSocketsClient |
| Python 中控 | Python 3.10+ | websockets、watchdog、openai、pynput |
| 前端 Dashboard | 任意编辑器 | 原生 HTML/CSS/JS，无框架依赖 |

### 6.3 通信架构

ESP32 通过 WiFi 与电脑在同一局域网内，使用 WebSocket 保持长连接。硬件每 2 秒上报一次传感器数据，服务器根据业务逻辑下发动作指令，前端通过另一个 WebSocket 连接实时接收状态推送。三者之间完全解耦，任意一端重启不影响其他模块。

## 七、人设与文案系统

### 7.1 角色设定卡

**基本信息：** 名称 DeskPal，性格傲娇毒舌但内心关心主人，自称"本大爷"或"本管家"，称呼主人为"主人"或"笨蛋主人"。

**语言模式：** 口癖包括"哼"、"才不是担心你"、"笨蛋主人"、"勉强夸你一下"。开心时用"本大爷"自称，语气轻快；生气时用命令语气说"罚你……"；表达关心时先否认再真诚，固定句式为"才不是……只是……"。

**情绪触发规则：**

| 情绪 | 触发条件 |
|------|---------|
| 开心 | 主人按时喝水、完成任务、摸它 |
| 生气 | 主人久不喝水、连续工作太久 |
| 委屈 | 被推开、被忽视太久 |
| 得意 | 巡视时、提醒成功时 |

### 7.2 完整文案库

**喝水提醒（三级递进）**

第一级（45 分钟未喝水，温和提醒）："喂，水都不喝的吗？你是骆驼？" / "本大爷特地跑过来提醒你喝水，感恩吧！" / "那边的水已经寂寞很久了，去陪陪它吧。"

第二级（60 分钟，语气升级）："说了几次了！喝水！现在！立刻！" / "哼，再不喝水我就一直挡着你的鼠标。" / "你的水杯在哭泣你知道吗？笨蛋主人。"

第三级（75 分钟，动真格）："够了够了！本大爷要占领你的工位了！" / "最后通牒！喝水！否则别想碰鼠标！" / "罚你！站起来去倒一杯新的热水！"

**休息提醒**

"眼睛不要钱的吗？休息一下啦。" / "本大爷允许你休息 5 分钟，快去。" / "才不是担心你近视……只是你盯屏幕的样子很蠢。" / "再不休息我就把你的鼠标叼走！"

**打招呼（主人到达）**

上午："哼，你终于来了，本大爷等很久了……才没有等你！" / "来了啊。今天也要在本大爷的监督下好好工作！" / "主人早！昨天的水喝够了吗？今天可别再让我催了。"

下午/返回："午睡回来了？精神点！下午也要好好干活！" / "还以为你不来了呢……才不是在等你。"

**任务完成**

"喂，你那个任务搞完了，快去看看。" / "任务完成了哦～才不是本大爷帮你盯着的，你自己看吧。" / "叮！完成了。本大爷的监工费什么时候结？"

**任务出错**

"那个……好像出错了。别看我，不是我弄的。" / "喂！报错了！快去处理！" / "嗯……有个坏消息。挂了。别慌，本大爷陪你。"

**被推开**

"呜……被推开了……才没有难过……" / "哼！推我！5 分钟后我还会来的！" / "过分！本大爷是为了你好！"

**喝水成功**

"哼，总算喝了。勉强夸你一下吧。" / "很好！继续保持！才不是在鼓励你呢。" / "终于想起来喝水了？本大爷的提醒功不可没！"

**巡视中**

"本大爷巡视一圈，看看有没有偷懒的。" / "巡逻中……一切正常。主人还在认真工作。" / "嗯嗯，桌面秩序良好。继续保持。"

**晚安/下班**

"要走了啊……明天也要按时来哦。才不是会想你。" / "今天辛苦了。本大爷准许你下班。晚安。" / "哼，终于舍得休息了。回去好好睡觉，别熬夜。"

## 十、演示策略

### 10.1 杀手级 Demo 场景

**场景 1："不喝水就堵你"（30 秒）**

演示者坐下工作 → 30 秒后（Demo 用缩短阈值） → 方块开始移动、跑向水杯 → OLED 显示生气脸 → 蜂鸣器响 → 电脑弹出"喂！水都不喝的吗！" → 演示者拿起水杯喝水 → 传感器检测到 → 方块回家 → OLED 显示开心脸 → 电脑显示"哼，总算喝了"

**场景 2："霸占鼠标"（20 秒）**

连续工作时间过长 → 方块跑到鼠标垫上趴着 → "哼，再不休息我就不走了" → 演示者推它 → 碰撞检测触发 → 委屈后退 → "呜...5 分钟后我还会来的"

**场景 3："任务播报"（15 秒）**

终端跑完一个编译任务 → 方块转圈庆祝 → "喂，你那个项目搞完了，快去看看" → 电脑弹出通知 + Dashboard 任务卡片变绿

### 10.2 演示技巧

- Demo 用阈值：把 45 分钟改为 30 秒，保证现场能快速触发
- 准备两套电池：防止 demo 中途没电
- 桌面铺白纸/纯色桌垫：让方块移动更明显
- 提前录制一段完整 demo 视频：现场翻车时可以播放

### 10.3 Pitch 结构（5 分钟）

前 30 秒用数据说明痛点（久坐一族的健康危机），接着 30 秒介绍 DeskPal 的一句话定位和产品形态。随后用 2 分钟现场演示三个核心场景。第 3 分钟快速过一遍系统架构图，展示软硬件结合的技术深度。第 4 分钟展示 AI 生成的傲娇日记，引发笑声和共鸣。最后 30 秒描绘未来拓展方向和商业化可能。

## 附录：快速启动指南

### 环境准备

Python 中控服务需安装 websockets、watchdog、openai、pynput 四个依赖库。ESP32 固件使用 Arduino IDE 开发，需安装 ESP32 开发板支持包，以及 Adafruit SSD1306 和 WebSocketsClient 两个库。前端 Dashboard 无需任何构建工具，直接用浏览器打开 HTML 文件即可运行。

### 启动顺序

先在电脑上启动 Python 中控服务，服务启动后会监听 WebSocket 端口。然后给 ESP32 上电，它会自动连接 WiFi 并与中控服务建立 WebSocket 连接。最后在浏览器打开 Dashboard 页面，即可看到实时状态。

### Demo 快速配置

Demo 演示时需将所有时间阈值大幅缩短，将正常的 45/60/75 分钟喝水提醒分别改为 30/60/90 秒，休息提醒从 50 分钟改为 60 秒，确保现场能在 1-2 分钟内快速触发所有演示场景。
